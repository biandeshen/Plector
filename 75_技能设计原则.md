# Plector 技能设计原则

> 版本：1.0
> 日期：2026-04-16
> 目的：确保所有技能能被 LLM 理解、组合、使用

关联：[[66_规范_技能开发规范]] | [[33_v1.4_SKILL规范实现]] | [[76_技能可发现性报告]]

---

## 核心理念

**技能是 LLM 的工具，不是互相调用的模块。**

LLM 通过 ReAct 循环看到所有技能的工具描述，决定何时组合使用。技能设计的核心目标是：**让 LLM 看了就知道什么时候用、怎么用、组合什么**。

---

## 一、技能设计四原则

### 原则 1：描述必须完整（≥50字）

描述 = 解决什么问题 + 典型场景 + 能力边界

**❌ 差：**
> "代码编写技能"

**✅ 好：**
> "代码编写技能。当用户需要创建新文件、修改现有代码、或进行批量代码替换时调用。能理解项目结构，支持相对路径操作，遵守项目根目录安全边界。典型场景：新建组件、重构、日志添加。"

### 原则 2：每个工具必须自描述

工具名即功能，描述说明"什么情况下用这个工具"。

**❌ 差：**
```json
{"name": "write_code", "description": "写入代码"}
```

**✅ 好：**
```json
{"name": "write_code", "description": "创建新文件或覆盖已有文件。当用户明确要求'写一个XX'、'创建XX文件'、或提供完整代码内容时调用。路径相对于项目根目录。"}
```

### 原则 3：必须包含触发词/场景说明

LLM 通过触发词匹配场景，格式：`Use when: ...` 或 `适用场景: ...`

**✅ 标准格式（放在 description 最后）：**
```
... 典型场景：XXX。
Use when: 用户要求... / 遇到...情况 / 适合...
```

**示例（code_writer）：**
```
代码编写技能，支持写入、读取、修改代码文件。
典型场景：新建组件、重构已有代码、批量替换、日志添加。
Use when: 用户说"写一个..."、"创建XX文件"、"帮我改这个函数"
```

### 原则 4：返回格式必须标准化

所有技能统一返回格式：

```python
{
    "success": True,           # bool，操作是否成功
    "data": {...},            # 实际返回数据
    "error": None             # str，错误信息（成功时为None）
}
```

## 二、skill.json 标准模板

```json
{
  "name": "skill_name",
  "version": "1.0.0",
  "description": "技能描述。典型场景：XXX。Use when: 用户说.../遇到...情况",
  "tier": "tier_2_enhanced",
  "tools": [
    {
      "name": "tool_name",
      "description": "工具描述。Use when: 用户说.../这个工具解决..."
    }
  ]
}
```

---

## 三、触发词规范

触发词放在 `description` 末尾，格式统一为：

```
Use when: 用户说"..." / 遇到...情况 / 适合...
```

**常见触发词模式**：

| 场景 | 触发词 |
|------|--------|
| 代码编写 | `用户说"写一个..."、"创建XX文件"、"帮我改代码"` |
| 代码审查 | `用户说"review"、"审查"、"看看有没有问题"` |
| 测试执行 | `用户说"跑测试"、"运行pytest"、"测试一下"` |
| 文件操作 | `用户说"列出文件"、"复制"、"移动"、"删除"` |
| 搜索 | `用户说"搜索..."、"查找..."、"有没有关于"` |
| 自改进 | `用户说"改进"、"优化"、"让它自己升级"` |
| 工作流 | `用户说"编排"、"组合这些技能"、"按流程执行"` |
| 健康检查 | `用户说"系统状态"、"CPU"、"内存"、"检查一下"` |

---

## 四、工具名命名规范

工具名必须：
- 动词开头（`write_`, `read_`, `search_`, `run_`, `check_`）
- 足够具体（`write_code` 而非 `write`）
- 保持一致（所有技能统一用 `_` 分隔）

---

## 五、与 LLM 的协作机制

技能不直接调用其他技能。协作通过 LLM 的 ReAct 循环实现：

```
用户任务
  ↓
LLM 理解任务 → 决定调用哪个技能/工具
  ↓
技能执行 → 返回结果
  ↓
LLM 观察结果 → 决定下一步（继续/换技能/组合）
```

**技能之间的关系由 LLM 决定，不是代码决定。**
