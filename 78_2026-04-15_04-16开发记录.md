## 四、流式响应（agent_loop.run_streaming）

### 架构

```
LLM token
  → agent_loop.run_streaming()
  → WebSocket 推送 chunk 事件
  → 浏览器逐字显示

事件类型：
- chunk: 单个 token
- toolExecuting: 工具执行中
- toolDone: 工具完成
- tool_call_start: 多工具调用开始
- done: 最终回复
```

### WebSocket handler

`channels/websocket.py` 已改用 `agent.run_streaming()`，推送流式事件

### LLM Client

`llm_client_v2._openai_stream()`: tool_call.arguments 必须 json 字符串，异常时 fallback 非流式

## 五、技能设计原则

文档：`docs/SKILL_DESIGN_PRINCIPLES.md`

### 核心原则

```
技能只是工具清单
LLM 才是指挥家
ReAct 循环 = LLM 看到任务 → 决定用什么 → 执行 → 观察 → 继续
```

### skill.json 必须包含

```json
{
  "name": "...",
  "triggers": ["什么场景用我"],
  "description": "我能做什么、输出什么",
  "events": ["我会发布什么事件"],
  "dependencies": ["我依赖哪些技能"]
}
```

### 不应该做的事

- ❌ 用 Python 代码写死执行逻辑
- ❌ 在 implementation.py 里写"先调A再调B"的硬编码流程
- ❌ 创建新的 Python 角色类（已有 external-skills/roles/）

### 应该做的事

- ✅ 纯 YAML 驱动：agency_orchestrator 工作流是 YAML
- ✅ skill.json 描述清楚"什么时候用"
- ✅ LLM 读取配置自己决定怎么组合
- ✅ 代码只做机械性的事（加载、执行、返回）

## 六、ReAct 元认知 + 技能主动联动

文档：`TASK.md`

### 决策规则

```
任务进来
  ↓
context_refresher 分析复杂度
  ↓
复杂 → agency_orchestrator 编排多角色
     → brainstorming 参与方案生成
     → external-skills/ 角色被调用
  ↓
简单 → 直接执行
```

### agency_orchestrator 角色库

174 个 YAML 角色在 `external-skills/roles/`，涵盖：
- engineering/ — 开发、架构、代码审查
- design/ — UI/UX、品牌、可视化
- strategy/ — 战略、剧本、运营
- 支持/ — 分析、总结、合规
- 测试/ — 性能、安全、无障碍
