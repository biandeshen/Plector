# BRD/PRD 精简思路说明

**日期**: 2026-04-27
**目的**: 解释重构思路，供审核参考

---

## 一、为什么要精简

### 1.1 原文档的问题

原 BRD v2.0 + PRD v1.6 存在三个核心问题：

**问题1：过度规划**
- 近千行文档中，70% 内容是 v2.x/v3.x 的详细规划
- 这些规划没有用户需求支撑，没有市场信号验证
- 本质上是在"竞品有什么我们抄什么"的逻辑下堆砌

**问题2：核心失焦**
- 大量竞品架构研究笔记（NanoBot/Hermes/DeerFlow）混入产品文档
- 这些是"研究性质"的，不是"决策性质"的
- 一个有效的产品文档应该只包含决策，不包含研究过程

**问题3：定位模糊**
- 声称"轻量级 + 企业"，但这两个词本身就是矛盾的
- 4 个核心卖点（可治理/可观测/可集成/可落地）可以描述任何企业软件
- 没有明确回答：Plector 到底是给谁用的？

### 1.2 精简原则

| 原则 | 含义 |
|------|------|
| **只写决策，不写研究** | 竞品分析保留结论，删除详细研究过程 |
| **只写 v1.x，不写 v2.x** | v2.x 是规划，不是产品 |
| **只写必要项，不写填充项** | 删除所有"看起来酷但不需要"的特性 |
| **定位清晰，不留歧义** | 一句话定位 + 明确目标用户 |

## 二、删除了什么，为什么

### 2.1 删除：竞品架构详细分析（~150行）

**原内容**：
- NanoBot/Hermes/DeerFlow 架构亮点详细分析
- Dream 两阶段记忆、SubagentManager、FTS5 等技术细节

**删除原因**：
1. 这些是"研究笔记"，不是"产品决策"
2. 复制粘贴竞品架构不等于理解竞品
3. 与 Plector 核心价值（轻量+可落地）关联度低

**保留什么**：
- 竞品对比表（1页）- 足够说明差异化

1. v2.x 详细方案，不是 v1.x 发布范围
2. 过度详细的设计细节在开发阶段会变化
3. 没有用户需求支撑

**保留什么**：

- LangGraph 状态机集成方案

2. 只是"看起来酷"的备选方案

**保留什么**：

## 三、保留了什么，为什么

### 3.1 保留：问题陈述

**原因**：
- 产品成功的起点是"解决真实问题"
- 明确目标用户和痛点，才能判断功能优先级
- 当前版本只保留了企业 AI 落地负责人这个明确角色

### 3.2 保留：竞品对比表

**原因**：
- 差异化定位是产品存在的理由
- 但只保留结论，不保留详细分析
- 一页表格足够，详细的竞品分析可以放在 research 目录

### 3.3 保留：MoSCoW 功能表

**原因**：
- 这是产品文档的核心
- 每个功能的状态标记（✅/🔄/📅）让读者一眼看清进度

### 3.4 保留：核心用户故事

**原因**：
- 用户故事是验证产品价值的基础
- 每个故事都有具体的验收标准
- 删除了所有没有验收标准的"规划性故事"

### 3.5 保留：发布计划

**原因**：
- 让团队和用户清楚知道接下来会发生什么
- 只保留 v1.x 的阶段性目标，不写 v2.x 详细规划

## 四、核心定位调整

### 4.1 原定位的问题

**原定位**："让复杂智能体在企业可落地"

**问题**：
- "轻量级"和"企业"是矛盾的
- 用户会问：到底给个人还是给企业？
- 4 个卖点可以描述任何企业软件

### 4.2 新定位

**一句话**：Plector = 企业级 Agent 的驾驶舱

**更清晰的含义**：
- 目标用户：企业 AI 落地负责人（不是个人开发者）
- 核心痛点：Agent 上线后无法治理/观测/自愈
- 解决方案：轻量底座 + 闭环自愈 + OTel 可观测性

## 六、后续建议

### 6.1 文档分层

建议将文档分为三层：

| 层级 | 文档 | 受众 |
|------|------|------|
| 决策层 | BRD/PRD（精简后） | 内部团队 |
| 研究层 | `docs/research/竞品分析.md` | 研究人员 |
| 参考层 | `docs/specs/legacy/` | 历史参考 |

### 6.2 验证假设

精简后的文档基于以下假设，需要在产品迭代中验证：

1. 企业 AI 落地负责人确实存在"无法治理/观测/自愈"的痛点
2. 5 分钟上手是真实需求，不是假设
3. 闭环修复成功率 ≥ 70% 是可实现的目标

### 6.3 下一步

1. 审核新文档是否清晰表达了产品意图
2. 如果通过，将现有文档移至 `legacy/` 目录
3. 在开发过程中更新新文档的状态标记
