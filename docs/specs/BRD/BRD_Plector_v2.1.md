# Plector 商业需求文档（BRD）v2.1

**版本**: 2.1
**状态**: 待审核
**日期**: 2026-04-27

---

## 1. 问题陈述

**目标用户**：企业 AI 落地负责人（IT 运维、DevOps、AI 团队负责人）

**核心痛点**：

| 痛点 | 现状 | 影响 |
|------|------|------|
| **无法治理** | Agent 上线后技能失控，不知道哪个在出错 | 生产事故频发 |
| **无法观测** | 没有链路追踪，出了问题不知道根因 | 排障困难 |
| **无法自愈** | 失败依赖人工介入，7x24 值守 | 运维成本高 |

**当前方案不足**：

| 方案 | 问题 |
|------|------|
| Dify/Coze | 全家桶，定制受限，学习曲线中等 |
| LangGraph/AutoGen | 功能强大但复杂度高（2-3周学习） |
| 自研 | 重复造轮子，治理能力要从零构建 |

---

## 2. 产品定位

**一句话定位**：Plector = 企业级 Agent 的驾驶舱

**核心价值**：
- **5分钟上手**（vs LangGraph 2-3周）
- **原生可观测性**（OpenTelemetry 标准输出）
- **闭环自愈**（技能健康分 + 自动修复）

**目标**：让中小团队 5 分钟内编排第一个企业智能体流程。

## 5. 商业模式

**GitLab 模式**：开源核心 + 企业增值

| 版本 | 功能 | 定价 |
|------|------|------|
| 开源核心 | 基础引擎 + 核心技能 + CLI | 免费 |
| 企业版 | RBAC + 审计日志 + SLA + 技术支持 | 单独报价 |

---

## 6. 成功指标

- **代码质量**：核心代码 < 5000 行，单元测试覆盖率 ≥ 80%
- **可用性**：安装后 5 分钟内可运行第一个示例
- **可靠性**：闭环修复成功率 ≥ 70%
- **社区**：发布后 3 个月 GitHub Stars 和贡献者数量

---

## 7. 风险与应对

| 风险 | 可能性 | 影响 | 应对 |
|------|--------|------|------|
| 技能治理过于复杂，用户不愿使用 | 中 | 中 | 提供默认策略，允许关闭 |
| 闭环修复失败导致事故扩大 | 中 | 高 | 人工通知 + 保留现场 |
| 事件总线性能瓶颈 | 中 | 高 | 支持 Redis 后端 |
