## 1. 产品背景与市场机会

### 1.1 市场痛点

2026年，AI Agent 市场已从"积木时代"迈向"系统集成时代"：

- **"框架时代"的局限**：LangGraph、AutoGen 等框架功能强大，但部署复杂、治理能力不足。企业在选型时发现，这些框架难以满足生产环境的**安全合规、审计追溯、系统集成**要求。
- **"升维竞争"的现实**：领先项目已向上下两端延伸——
  - OpenClaw：Gateway-Hub-Agents 架构，跨越应用成为数字助理（25万+ GitHub Stars）
  - DeerFlow 2.0：字节跳动主导，"超级智能体平台"，隔离环境完成复杂长周期任务
  - CrewAI：企业级多智能体平台，60% 美国财富500强客户，强安全审计壁垒
- **"可落地"的缺口**：仅凭"自我进化"已不足以构成护城河。竞争已从技术特性，转向**生态系统、治理能力、工程化落地**。

### 1.2 行业趋势

- **从"用框架"到"搭系统"**：企业不再选择框架，而是寻找稳定、可治理的系统。
- **安全治理成为生命线**：企业级 Agent 必须有严格的安全边界、不可篡改的审计日志、细粒度权限控制。
- **可观测性是标配**：遵循 OpenTelemetry 标准，输出可关联的 Trace、Metrics、Logs，集成 APM 系统。
- **轻量级切入更务实**：大型框架学习曲线陡峭（2-3周），中小团队需要"开箱即用+低代码编排"的轻量方案。

### 1.3 战略定位调整

| 旧定位（v1.x） | 新定位（v2.0） |
|-------------|---------------|
| "自我进化的极简 AI Agent 引擎" | **"让复杂智能体在企业可落地"** |
| 兼顾个人/团队/企业 | 聚焦**中小团队/企业内部门** |
| 技术特性驱动 | **治理能力+工程化落地** 驱动 |
| "自我进化"为核心卖点 | **"可落地"为核心命题，自我进化为底层支撑** |

**一句话定位**：Plector 是让复杂智能体在企业可落地的轻量级底座。

## 3. 产品核心功能（BRD 层面）

### 3.1 核心命题：从"自我进化"到"可落地"

**新核心命题**：让复杂智能体在企业可落地

| 旧命题 | 新命题 | 说明 |
|--------|--------|------|
| "自我进化" | **可治理** | 技能健康分、依赖检测、自动淘汰 |
| "极简" | **可观测** | OpenTelemetry 标准，Trace/Metrics/Logs |
| "事件驱动" | **可集成** | 中间件链、MCP 原生、开放 API |
| "闭环自愈" | **可落地** | 安全边界、审计日志、企业部署 |

### 3.2 极简内核（保留）

- **自主决策循环**：Agent 能够自主推理、调用工具、观察结果并迭代。
- **稳定工具调用**：支持将任意 Python 函数暴露为可调用工具，并保证调用格式的可靠性。
- **事件驱动架构**：组件之间通过异步事件解耦，支持技能协作和闭环触发。
- **动态上下文**：根据项目配置文件自动构建系统提示词，实现人格化和项目记忆。

### 3.2 自我进化闭环

- **自愈流程**：系统能够定义条件触发的自动修复流程（例如测试失败后自动诊断、修复、重测）。
- **技能治理**：对核心能力进行健康度评估、依赖检测和自动淘汰，防止技能膨胀。
- **自我审视**：定期扫描系统状态，生成依赖关系、事件完整性和健康报告。
- **经验学习**：从成功或失败的调用中提取模式，优化后续决策。

### 3.3 技能与工具分层

| 类型 | 定义 | 示例 | 治理 |
|------|------|------|------|
| **核心技能** | 有状态、有事件、有依赖 | 健康监控、错误知识化、自学习 | 完整治理（健康分、淘汰） |
| **工具函数** | 无状态、无事件、无依赖 | 格式转换、网络搜索 | 无治理，仅注册 |

核心技能数量严格控制在 10-15 个，其余能力通过轻量工具函数扩展。

### 3.4 多渠道接入

- **命令行界面**：支持交互式对话和单次查询。
- **WebSocket**（可选）：实时双向通信，用于 Web 界面。
- **HTTP API**（远期）：供外部系统调用。

## 11. 灵魂文件系统设计

### 11.1 设计理念

**核心思想**：让 AI Agent 拥有"灵魂"——人可读、人可写、AI 可理解的文件化记忆系统。

参考 OpenClaw 的 SOUL.md/USER.md/AGENTS.md 设计，Plector 将构建分层灵魂文件系统。

### 11.2 文件结构

```
Plector/
├── soul.md              # AI 人格定义（用户编辑）
├── user.md              # 用户画像（用户编辑）
├── agents.md            # 安全规则与工具权限（用户编辑）
├── memory.md            # 长期知识记忆（AI + 用户共同维护）
├── memory/              # 记忆目录
│   ├── 2026-04-16.md    # 每日日志（AI 自动写入）
│   └── sessions/        # 会话存档
└── core/
    ├── soul_loader.py   # 灵魂文件加载器
    └── memory_writer.py # 记忆自动写入器
```

### 11.3 灵魂文件定义

| 文件 | 职责 | 修改权限 |
|------|------|---------|
| **soul.md** | AI 人格、沟通风格、能力边界 | 仅用户（除非授权 AI） |
| **user.md** | 用户姓名、职业、技术栈、偏好 | 仅用户（除非授权 AI） |
| **agents.md** | 安全边界、工具权限、记忆权限 | 仅用户 |
| **memory.md** | 项目知识、技术决策、用户习惯 | AI + 用户 |
| **memory/YYYY-MM-DD.md** | 每日日志、短期记忆 | AI 自动 |

### 11.4 实施计划

| 阶段 | 任务 | 优先级 |
|------|------|--------|
| **Phase 1** | 创建 soul.md/user.md/agents.md 模板 | P0 |
| **Phase 1** | 实现 soul_loader.py 热加载 | P0 |

## 12. 安全治理与可观测性

### 12.1 安全治理架构

**核心原则**：安全是企业用户的生命线，Plector 必须内置企业级安全能力。

| 安全维度 | 设计方案 | 优先级 |
|----------|----------|--------|
| **输入安全** | 输入清理与验证，防止提示注入 | P0 |
| **权限控制** | RBAC/ABAC 细粒度权限管理 | P0 |
| **执行隔离** | 工具沙箱，防止危险系统调用 | P0 |
| **审计日志** | 不可篡改的操作审计日志 | P0 |
| **输出安全** | 输出内容过滤与脱敏 | P1 |

### 12.2 中间件链设计

```
执行顺序：Logging → Security → Governance → Memory → SkillChain
```

| 中间件 | 功能 | 说明 |
|--------|------|------|
| **LoggingMiddleware** | 请求/响应日志 | P0 |
| **SecurityMiddleware** | 输入验证、权限检查 | P0 |
| **AuditMiddleware** | 不可篡改审计日志 | P0 |
| **GovernanceMiddleware** | 技能健康分监控 | P1 |
| **MemoryMiddleware** | 结果自动记忆 | P1 |
| **ObservabilityMiddleware** | OpenTelemetry 集成 | P0 |

### 12.3 可观测性标准

> **安全合规官建议（OTel 升级为 P0）**：可观测性是企业级用户选型的必要条件，OTel 必须作为 P0 阻断优先级。

遵循 **OpenTelemetry** 标准，输出可关联的 Trace、Metrics、Logs：

| 可观测性维度 | 实现 | 企业集成 | 优先级 |
|-------------|------|----------|--------|
| **Traces** | 分布式链路追踪 | Jaeger/Zipkin | **P0** |
| **Metrics** | Agent 性能指标 | Prometheus/Grafana | **P0** |
| **Logs** | 结构化日志（JSON Lines） | ELK/Loki | **P0** |
| **Spans** | 技能执行时长、工具调用延迟 | 链路可视化 | **P0** |
| **APM 集成** | 应用性能监控 | Datadog/New Relic | P1 |

**OTel MVP 实现范围**（运维建议）：
- Docker Compose 一键部署（Prometheus + Grafana + Loki）
- K8s 配置模板（Deployment + Service + ConfigMap）
- 闭环指标 Dashboard（修复成功率、执行耗时、失败原因分布）
- 热更新触发指标（文件哈希变化检测 + 自动重载）
