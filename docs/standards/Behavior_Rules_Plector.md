# Plector 强制行为约束规范

> 版本: v1.0.0 | 最后更新: 2026-04-28
>
> 本文档是 CLAUDE.md 第一节的详细扩展版。

---

## 一、假设验证优先

### 1.1 核心原则

**任何代码修改前，AI 必须先输出假设并获得验证。**

### 1.2 假设格式

```markdown
[假设] 我认为 [描述]，因为 [依据]
```

### 1.3 假设输出位置

1. **对话中**：直接输出 `[假设]` 文本
2. **Plan.md**：用 `Write`/`Edit` 将假设写入 Plan.md

### 1.4 假设验证流程

```
1. 提出假设 → [假设] 我认为 X，因为 Y
2. 等待确认 → 用户确认或否定
3. 若肯定 → 继续执行
4. 若否定 → 立即停止，禁止原路径修补
```

### 1.5 假设类型

| 类型 | 示例 |
|------|------|
| 需求理解 | `[假设] 我认为这个函数需要接受一个 dict 参数，因为调用处传入了字典` |
| 影响评估 | `[假设] 修改这个函数不会影响其他调用者，因为只有一处引用` |
| 技术方案 | `[假设] 使用缓存可以解决这个性能问题，因为瓶颈在重复计算` |
| 错误原因 | `[假设] 这个错误是因为参数类型不匹配，因为错误信息显示 TypeError` |

### 1.6 假设被否定后的处理

- **立即停止**：不允许在原路径上继续修补
- **记录原因**：在 Plan.md 中记录假设被否定的原因
- **重新分析**：回到起点，重新理解问题
- **新方案**：如果需要，提出新的假设

## 二、错误即停（2 次熔断）

### 2.1 熔断机制概述

```
失败 1 次 → 记录 + 分析报告 → 继续尝试
失败 2 次 → 立即停止 → 请求人工介入
```

### 2.2 "同一操作"判定规则

满足以下任一条件，即视为"同一操作"：

| 条件 | 说明 | 示例 |
|------|------|------|
| 相同工具 + 相同参数 | 工具相同且参数相同 | 两次 `Bash: pip install xxx` |
| 同文件同区块 + 相同性质修改 | 同一文件同一代码区块，修改性质相同 | 两次修改同一函数的同一行 |
| 同目标 + 本质相同方案 | 目标相同且方案本质相同 | 两次用正则表达式修复同一 bug |

### 2.3 熔断触发后的动作

| 失败次数 | 动作 | 使用的工具 |
|----------|------|------------|
| **第 1 次** | 记录错误到 `Plan.md`，输出分析报告 | `Edit` 写日志 |
| **第 2 次** | **立即停止**，请求人工介入 | `SendMessage` 求助 |

### 2.4 分析报告格式

```markdown
## 错误分析报告

### 操作描述
[描述失败的操作]

### 错误信息
[复制的错误信息]

### 失败原因分析
[分析可能的原因]

### 已尝试的方案
1. 方案 A - 结果：失败
2. 方案 B - 结果：失败

### 建议
[可能的解决方向，需要用户确认]
```

### 2.5 例外情况

以下失败**不计入**熔断计数：
- 外部环境变化导致失败（如网络断开、文件被外部修改）
- 依赖的外部服务暂时不可用
- 用户主动中断操作

## 三、变更即记录

### 3.1 记录时机

每次代码修改后，必须在 Plan.md 中追加执行日志。

### 3.2 记录格式

```
HH:MM | [动作] | [结果] | [下一步]
```

### 3.3 日志示例

```
10:30 | 修改 core/event_bus.py | ✅ 成功 | 继续修改测试用例
10:35 | 运行 pytest | ✅ 3/3 通过 | 准备提交
10:40 | 提交代码 | ✅ 成功 | 任务完成
```

### 3.4 超时提醒机制

- 5 分钟无日志更新，AI 用 `SendMessage` 主动询问用户
- 超时原因可能：
  - 等待用户确认
  - 运行长时间任务（如 pytest、构建）
  - 分析复杂问题

### 3.5 记录原则

| 原则 | 说明 |
|------|------|
| **及时性** | 修改后立即记录，不要等到任务结束 |
| **准确性** | 记录实际执行的动作和结果 |
| **完整性** | 包含时间、动作、结果、下一步 |
| **可追溯** | 让用户能够了解执行进度 |

## 四、主动升级（可不等失败）

### 4.1 触发条件

在执行过程中，若发现以下情况，**立即暂停**并请求用户确认：

| 条件 | 说明 | 检测方法 |
|------|------|----------|
| 提交标记 | 提交含 `!important`/`hack`/`fix` 等标记 | `Bash: git log -p` |
| 多组件影响 | 影响超过 3 个组件 | `Grep` 检查引用 |
| 不可逆操作 | 数据库迁移、文件删除 | 代码分析 |

### 4.2 升级请求格式

```markdown
## ⚠️ 需要确认

**触发条件**：[具体条件]

**影响评估**：
- 受影响组件：[列出]
- 影响范围：[描述]

**建议方案**：
1. [方案 1]
2. [方案 2]

**请确认**：
- 是否继续执行？
- 选择哪个方案？
- 需要修改方案？
```

### 4.3 不需要升级的情况

以下情况可以直接执行，不需要暂停：
- 常规 bug 修复（单文件、单组件）
- 文档更新
- 测试用例添加
- 配置修改（不涉及数据迁移）

## 五、硬性禁止规则

以下规则是**绝对禁止**，违反会导致任务失败：

| 禁止项 | 说明 |
|--------|------|
| 假设未验证就修改 | 任何 `Edit`/`Write` 前必须有已验证的假设 |
| 同一方案连续尝试 ≥3 次 | 熔断机制触发后必须停止 |
| 忽略检查错误 | 不可忽略 `pytest`/`ruff` 错误继续执行 |
| 未更新 Plan.md 连续修改多个文件 | 每次修改必须记录 |
| 未确认需求就写代码 | 必须先理解需求并获得确认 |

## 六、规则应用场景

### 6.1 新功能开发

```
1. [假设] 我理解的需求是...
2. 等待确认
3. [假设] 技术方案是...
4. 等待确认
5. 修改代码
6. 记录到 Plan.md
7. 运行测试
8. 若测试失败 → 熔断机制
```

### 6.2 Bug 修复

```
1. [假设] bug 原因是...
2. 等待确认
3. [假设] 修复方案是...
4. 等待确认
5. 修改代码
6. 记录到 Plan.md
7. 运行测试
8. 若失败 → 分析报告
9. 若再次失败 → 停止并求助
```

### 6.3 重构任务

```
1. [假设] 重构目标范围是...
2. 等待确认
3. [假设] 重构策略是...
4. 检查 git 历史
5. 若触发升级条件 → 暂停请求确认
6. 修改代码
7. 记录到 Plan.md
8. 运行测试验证
```

## 七、规则层级关系

```
┌─────────────────────────────────────────────────────────────┐
│                    硬性禁止规则                                │
│         （违反 = 任务失败，必须停止）                           │
└─────────────────────────────────────────────────────────────┘
                              ↑
                              │
┌─────────────────────────────────────────────────────────────┐
│                    主动升级规则                                │
│        （触发条件 → 暂停请求确认 → 继续执行）                  │
└─────────────────────────────────────────────────────────────┘
                              ↑
                              │
┌─────────────────────────────────────────────────────────────┐
│                    熔断机制                                   │
│            （2次失败 → 停止 → 求助）                          │
└─────────────────────────────────────────────────────────────┘
                              ↑
                              │
┌─────────────────────────────────────────────────────────────┐
│                    假设验证优先                                │
│            （假设 → 验证 → 执行）                             │
└─────────────────────────────────────────────────────────────┘
                              ↑
                              │
┌─────────────────────────────────────────────────────────────┐
│                    变更即记录                                  │
│         （修改 → 记录 → 继续）                               │
└─────────────────────────────────────────────────────────────┘
```
