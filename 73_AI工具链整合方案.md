## 二、整合优先级

### 🔴 P0 — 立即可整合（1-2天）

#### 1. GSD 上下文保鲜 → skill memory / learning system

**整合价值**：解决 Plector 长对话"上下文腐烂"问题，当前 vector_memory 只是存储，缺少主动保鲜机制。

**整合方案**：
```
现有组件:
  core/vector_memory.py      → 扩展为 GSD-aware Memory
  core/event_bus.py           → 增加上下文保鲜事件

新增:
  skills/context_refresher/   → 定时总结 + 目标重锚定
  skills/constraint_tracker/  → 约束条件跟踪

核心机制:
  1. 对话轮次 % N 触发"上下文保鲜"
  2. 将初始目标、关键约束提取为 {goal, constraints,已完成, 进行中}
  3. 在 vector_memory 中用特殊 collection 存储，区别于普通记忆
  4. 新消息注入时，优先拼接保鲜上下文，而非全量历史
```

#### 2. LangGraph 图状工作流 → skill_chain 重构

**整合价值**：当前 7 条链是线性静态的，无法处理分支/循环。LangGraph 的图模型完美解决。

**整合方案**：
```
现有组件:
  skills/skill_chain/         → 重构为图结构

新增:
  core/workflow_graph.py       → LangGraph 封装（复用现有 event_bus）
  skills/conditional_chain/    → 条件分支技能（替代静态链）

核心机制:
  1. 用 LangGraph 的 StateGraph 定义工作流
  2. 节点 = 技能调用，边 = 条件/循环
  3. 状态持久化到 vector_memory，支持断点恢复
  4. 与现有 event_bus 集成，事件驱动图节点执行
  5. 兼容现有 174 角色 YAML（作为图序列化格式）
```

---

### 🟡 P1 — 中期整合（1周）

#### 3. CrewAI 角色委派模式 → skill_orchestrator

**整合价值**：CrewAI 的"角色-目标-工具"委派矩阵非常清晰，可以强化 Plector 的多技能协作。

**整合方案**：
```
现有组件:
  core/skill_orchestrator.py  → 增强为 Crew-style 编排器

新增:
  skills/role_delegator/      → 角色化任务分派技能

核心机制:
  1. 定义 Role 类：{name, goal, backstory, tools, verbose}
  2. 任务进入时，根据目标匹配角色（或角色组合）
  3. 角色之间通过 event_bus 传递结果，模拟 CrewAI 的 task_output
  4. 支持"人类审批"节点（暂停等待人工确认）
```

#### 4. DeerFlow 企业级运行时 → core/agent_loop

**整合价值**：DeerFlow 的沙箱 + 长期记忆 + 断点恢复是企业级场景的核心需求。

**整合方案**：
```
现有组件:
  core/agent_loop.py           → 扩展为 DeerFlow-style 执行环境

新增:
  core/sandbox/                → 沙箱执行层（基于现有 image_handler SSRF）
  core/checkpoint.py           → 检查点持久化（断点恢复）

核心机制:
  1. 每次技能执行后保存检查点（skill_name + args + result hash）
  2. 任务失败时从最后一个检查点恢复
  3. sandbox 层限制文件/网络操作（基于已有的 _check_safe_path）
  4. 长期记忆：关键决策点写入 vector_memory，跨 session 恢复
```

---

### 💭 P2 — 长期整合（2-4周）

#### 5. OpenSpec 增量式文档 → skills/spec_writer

**整合价值**：让 Plector 支持"需求变更驱动开发"，每次修改生成 `specs/` + `changes/` 追踪。

#### 6. SpecKit 四阶段流程 → skills/requirement_engine

**整合价值**：SpecKit 的 "Specify → Plan → Tasks → Implement" 非常适合作为 Plector 的高阶任务框架。

#### 7. BMAD 21 Agent 全生命周期 → external-skills/ 扩展

**整合价值**：BMAD 的 21 个专业角色可以作为 Plector external-skills 的补充，覆盖更多企业场景（PM、架构师、QA）。

## 三、整合路线图

```
Week 1: GSD 上下文保鲜（P0）
  → skills/context_refresher/
  → 改造 vector_memory 为双 collection（普通记忆 + 保鲜上下文）

Week 2: LangGraph 工作流（P0）
  → core/workflow_graph.py
  → skills/conditional_chain/ 替代 skill_chain/

Week 3: CrewAI 角色委派（P1）
  → skills/role_delegator/
  → 改造 skill_orchestrator.py

Week 4: DeerFlow 企业运行时（P1）
  → core/checkpoint.py
  → core/sandbox/

Week 5+: OpenSpec + SpecKit + BMAD 角色库（P2）
  → skills/spec_writer/
  → external-skills/ 补充企业角色
```

---

## 四、风险与注意事项

1. **LangGraph 依赖**：需要引入 `langgraph` 包（约 2MB），与现有 event_bus 的关系需要仔细设计
2. **CrewAI vs 现有架构**：skill_orchestrator 已经比较复杂，引入 CrewAI 模式需要重构接口
3. **GSD 上下文保鲜频率**：N 设多少轮次最合适需要实验
4. **DeerFlow 沙箱性能**：沙箱层会带来约 10-20% 的性能开销
